    3. Convert the f-string HTML to a Jinja2 template `app/web/templates/base.html`:

    **Conversion rules (CRITICAL -- f-string to Jinja2 escaping is the hardest part):**
    - f-string `{variable}` -> `{{ variable }}` (Jinja2 interpolation)
    - f-string `{{` (escaped literal brace for CSS/JS) -> `{` (Jinja2 treats as literal)
    - f-string `}}` (escaped literal brace) -> `}` (Jinja2 treats as literal)
    - JavaScript `${...}` template literals -> stay as-is (Jinja2 ignores `${}`)
    - CSS `{ }` braces -> stay as-is (Jinja2 only interprets `{{ }}` and `{% %}`)

    **Template context variables:** The route handler passes `data` (dict) and `data_json` (pre-serialized JSON string). The template should use:
    - `{{ data_json | safe }}` for injecting the JSON data into the `<script>` block (the `| safe` filter prevents HTML-escaping the JSON)
    - Any server-side rendered content should use `{{ data.field_name }}`

    **Key pattern:** The current f-string builds `data_json = json.dumps(data).replace("</", r"<\/")` at line 60 of presentation.py, then injects it into a `<script>` tag as `const DATA = {data_json};`. In the Jinja2 template this becomes `const DATA = {{ data_json | safe }};`. The `data_json` is already escaped by Python before passing to the template.

    **Approach for the massive f-string:**
    - Read the entire `render_dashboard` function
    - The f-string starts with `<!doctype html>` and contains inline CSS, HTML structure, and inline JavaScript
    - Most of the content is static HTML/CSS/JS with a few `{variable}` interpolations
    - The main dynamic content is the `{data_json}` injection in the script tag
    - Some HTML sections use Python f-string expressions like `{len(data.get('games', []))}` -- convert these to Jinja2: `{{ data.games | length }}`
    - For conditional blocks using Python ternary in f-strings, convert to `{% if %}...{% endif %}`
