  <files>app/data/database.py, app/web/app.py, app/core/service.py, tests/web/test_cron_endpoints.py</files>
  <behavior>
    - POST /api/cron/settle without Authorization header returns 401
    - POST /api/cron/settle with wrong Bearer token returns 401
    - POST /api/cron/settle with valid CRON_SECRET calls settle_outstanding() and returns JSON result
    - POST /api/cron/refresh with valid CRON_SECRET triggers model refresh and returns JSON result
    - Both endpoints log to cron_log table (timestamp, task, status, details, duration_ms)
    - On settle failure, retry once after 5 min delay is documented but not tested (async behavior)
    - Data source errors are logged with source name in details field
    - Polymarket 422 errors are caught and silenced (not logged as errors)
  </behavior>
  <action>
    1. In app/data/database.py, add cron_log table to _SCHEMA_SQL:
       ```sql
       CREATE TABLE IF NOT EXISTS cron_log (
           id          INTEGER PRIMARY KEY AUTOINCREMENT,
           run_at      TEXT    DEFAULT (datetime('now')),
           task        TEXT    NOT NULL,
           status      TEXT    NOT NULL,
           details     TEXT,
           duration_ms INTEGER
       );
       CREATE INDEX IF NOT EXISTS idx_cron_log_task ON cron_log (task);
       ```

    2. Add methods to TrackerDatabase:
       - log_cron_run(task: str, status: str, details: str, duration_ms: int) -> int
       - get_last_cron_run(task: str) -> dict | None (returns most recent row for given task)
       - get_cron_history(task: str | None, limit: int = 50) -> list[dict]

    3. In app/web/app.py, add a helper function _verify_cron_secret(request: Request) that checks Authorization header against os.environ.get("CRON_SECRET"). Returns True/False.

    4. Add POST /api/cron/settle endpoint:
       - Verify CRON_SECRET (return 401 JSON if invalid)
       - Call settle_outstanding() with timing
       - Log result to cron_log via TrackerDatabase
       - Return JSON with status, settled count, duration_ms
       - On exception: log as failed, return 500 with error details

    5. Add POST /api/cron/refresh endpoint:
       - Verify CRON_SECRET (return 401 JSON if invalid)
       - Call build_market_snapshot() with default TrackerConfig to trigger fresh data fetch
       - This refreshes the in-memory cache and validates data sources are responding
       - Log result to cron_log
       - Return JSON with status, data_sources checked, duration_ms
       - Catch Polymarket errors (422 specifically) and silence them (log.debug, not log.error)

    6. In app/core/service.py, wrap the existing Polymarket fetch in build_market_snapshot() to catch HTTPError 422 specifically and log at debug level instead of warning. This silences the known broken Polymarket API.

    7. Write tests in tests/web/test_cron_endpoints.py using FastAPI TestClient:
       - test_settle_no_auth_returns_401
       - test_settle_wrong_token_returns_401
       - test_settle_valid_token_runs_settlement (mock settle_outstanding)
       - test_refresh_valid_token_runs_refresh (mock build_market_snapshot)
       - test_cron_result_logged_to_db
       - test_polymarket_422_silenced
